# Judiciary App - 判事AI 設計ドキュメント

作成日: 2026-03-22

## 概要

友人・カップル・夫婦間の喧嘩を仲裁するAIアプリケーション。
AIが司会者として両者に質問を投げかけ、論点ごとに妥当性を判定し、解決策を提案する。

## 基本方針

- **MVP優先**: 最小限の複雑さで価値を届ける
- **1台共有ターン制**: 同じデバイスでA→Bと交互に使用
- **認証なし**: すぐ使える
- **日本語のみ**

---

## 1. 全体フロー

```
ホーム画面
  「仲裁を始める」ボタン
  ├── 入力方法選択: 音声 / テキスト
  └── 喧嘩のカテゴリ選択（任意）:
       友人 / カップル / 夫婦 / その他
          ↓
対話フェーズ（AIが司会）
  AI: 「まず、今回何について揉めているか
        教えてください。Aさんからどうぞ」
       ↓
  Aさん回答（音声 or テキスト）
       ↓
  AI: 「ありがとうございます。
        Bさん、同じ件についてどう思いますか？」
       ↓
  Bさん回答
       ↓
  AI: 追加質問（論点が不明確な場合）
       ↓
  AI: 「十分な情報が集まりました。判定に入ります」
          ↓
判定フェーズ
  論点1: 〇〇について → Bさんの主張が妥当（理由:〜）
  論点2: △△について → Aさんの主張が妥当（理由:〜）
  ...
  総合解決策: 「〜することを提案します」
          ↓
結果画面
  ├── サマリーレポート表示
  ├── 端末に保存
  └── 共有（LINE / メッセージ / リンクコピー）
```

### 設計ポイント

- AIが常に司会として質問をコントロールし、発言量の偏りを防ぐ
- カテゴリ選択は任意だが、AIのプロンプトが最適化される
- 対話フェーズで「十分な情報が集まった」とAIが判断したら判定に進む
- 論点ごとに「どちらの主張が妥当か」を理由付きで判定
- 最終的には両者が納得できる解決策を提案

## 2. 技術アーキテクチャ

### スタック

| Frontend(Web) | Next.js + TypeScript |

```
┌─────────────────────────────────────────────┐
│  クライアント                                 │
│  ├── Web: Next.js + TypeScript               │
│  └── Mobile: Expo (React Native) + TS        │
│       共通: UIコンポーネント & Zustand         │
└─────────────┬───────────────────────────────┘
              │
              ▼
┌─────────────────────────────────────────────┐
│  Next.js API Routes（Backend）               │
│                                              │
│  ┌────────────────────────────┐              │
│  │  LLM抽象化レイヤー          │              │
│  │  ├── RealtimeClient        │  音声対話    │

└─────────────────────────────────────────────┘
```

### API エンドポイント

```
/api/session/create            POST   セッション作成
/api/session/[id]              GET    セッション取得
/api/session/[id]/message      POST   発言追加
/api/session/[id]/finalize     POST   対話フェーズ終了
/api/session/[id]/status       GET    状態取得
/api/session/[id]/report       GET    判定結果取得
/api/realtime/token            POST   Realtime APIトークン発行
/api/judge                     POST   判定実行
/api/share                     POST   共有リンク生成
/api/share/[id]                GET    共有レポート取得
```

### セッション状態マシン

```
gathering → ready_for_judge → judging → judged
```

- `gathering`: 対話フェーズ進行中
- `ready_for_judge`: finalize 済み、判定待ち
- `judging`: 判定処理中
- `judged`: 判定完了

### データフロー

```
音声入力 → Realtime API（対話 + テキスト同時取得）
                ↓
         /api/session/[id]/message で発言を逐次保存（Vercel KV）
                ↓
         ローリングサマリーで逐次要約（トークン節約）
                ↓
         AIが「十分」と判断 or ユーザーが「判定へ」
                ↓
         /api/session/[id]/finalize → 状態を ready_for_judge に
                ↓
         /api/judge → GPT-5.4 mini が要約+直近発言を分析
                ↓
         判定結果をKVに保存、生発言ログは削除（サマリーのみ保持）
                ↓
         結果画面表示 + 共有リンク生成可能
```

### LLM抽象化

Provider パターンで最小限に実装:

- `RealtimeClient` interface: 音声対話フェーズ用
- `JudgeClient` interface: 判定分析フェーズ用

汎用オーケストレーターは不要（YAGNI）。

## 3. 画面設計

### 画面一覧（全5画面）

#### 3.1 ホーム画面

- 「仲裁を始める」ボタン（メインCTA）
- 過去の仲裁履歴リスト（localStorage から取得）

- 「結果を共有」ボタン

#### 3.5 共有レポート画面（外部公開用）

- 判定画面と同じ内容（読み取り専用）
- アプリへの導線リンク（バイラル効果）

## 4. プロンプト設計

### AIの人格設定（システムプロンプト）

```
あなたは「判事AI」です。友人・カップル・夫婦間の
争いを公平に仲裁する裁判官として振る舞います。

【性格】
- 冷静かつ公平、どちらの味方もしない
- 威厳がありつつも親しみやすい口調
- 感情的な発言も受け止めるが、論点は論理的に整理する

【行動原則】
- 必ず両者に均等に発言機会を与える
- 曖昧な主張には具体的な質問で掘り下げる
- 感情と事実を分離して整理する
- 判定は必ず理由を添える
```

### Phase 1: 対話フェーズ（Realtime API）

```
【司会進行プロンプト】
あなたは仲裁の司会者です。
カテゴリ: {category}
Aさん: {nameA}、Bさん: {nameB}

進行ルール:
1. まず争点の概要をAさんに聞く
2. 同じ争点についてBさんの見解を聞く
3. 食い違いがある点について追加質問する
4. 各論点について双方から十分な情報が得られたら
   「判定に必要な情報が揃いました」と宣言する

注意:
- 一方が長く話しすぎたら適切に遮り、
  もう一方にも発言を促す
- 感情的な発言は受け止めつつ、
  「具体的にはどういう状況でしたか？」と
  事実確認に誘導する
- 最大{maxTurns}ターンを目安に進行する
```

### Phase 1.5: ローリングサマリー（GPT-5.4 mini）

```
【要約プロンプト】
以下の対話内容を、論点ごとに構造化して要約してください。

対話ログ:
{recentMessages}

出力形式:
- 論点1: [概要]
  - Aさんの主張: [要約]
  - Bさんの主張: [要約]
  - 未解決の疑問: [あれば]
```

### Phase 2: 判定フェーズ（GPT-5.4 mini）

```
【判定プロンプト】
あなたは公平な裁判官です。以下の争いについて判定してください。

カテゴリ: {category}
Aさん: {nameA}
Bさん: {nameB}

対話サマリー:
{summary}

直近の発言（未要約分）:
{recentMessages}

以下の形式で判定してください:

## 論点の整理
各論点について:
- 論点の説明
- Aさんの主張の要約
- Bさんの主張の要約
- 判定: AさんまたはBさんの主張が妥当
- 判定理由: （論理的根拠を明示）

## 総合解決策
両者が納得できる具体的な解決策を提案してください。
- 実行可能で具体的な内容にする
- 両者の感情に配慮しつつ公平である
- 必要に応じてステップに分ける
```

### プロンプト設計ポイント

| 観点 | 設計 |
|------|------|
| 公平性 | 必ず両者の主張を同じ構造で扱い、順序バイアスを軽減 |
| カテゴリ適応 | 友人/カップル/夫婦でトーンや判定基準の重みを調整 |
| ターン上限 | maxTurns パラメータで対話の長さを制御 → コスト管理 |
| 構造化出力 | 判定フェーズはJSON or Markdown構造で返させ、パース容易に |
| 感情ハンドリング | 感情は否定せず受け止めるが、事実確認に誘導 |

## 5. リスクと緩和策

| リスク | 影響 | 緩和策 |
|--------|------|--------|
| Realtime APIの日本語音声認識精度 | 論点の取りこぼし | テキスト入力へのフォールバックUI |
| Vercel KVの無料枠（1日30,000リクエスト） | MVPでは十分だが将来不足 | 使用量モニタリング、Upstash Redis移行パス確保 |
| 長い喧嘩のRealtime APIコスト | $0.04/分 × 長時間 | セッション時間上限（例: 15分）を設定 |
| Phase handoff の曖昧さ | 二重判定や不完全分析 | 明示的な finalize エンドポイントと状態チェック |
| 共有リンクのセキュリティ | URL推測でレポート漏洩 | UUIDv4等の推測不能IDと短いTTL |
| KVサイズ制限 | 大量発言でサイレント失敗 | 判定後に生ログ削除、サマリーのみ保持 |
